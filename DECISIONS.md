# WatchRoulette — engineering decision log

This is a narrative companion to `spec.md` and `TRACKER.md`, not a replacement for either. Those two answer "what did we decide" and "what's actually built." This one answers a different question: **why** — the problem as it was actually found, the options that were on the table, what got picked, and what that reasoning cost or saved later. Written for two audiences at once: future-me picking this project back up, and anyone (including an interviewer) who wants to see how the decisions actually got made, not just the polished result.

Chronological, roughly matching the module order in `spec.md`.

---

## Foundational calls (Modules 0–3)

**Backend: Supabase, no hand-built server.** The differentiated part of this app is the spin mechanic and the social loop around it, not a custom API layer. Supabase's auto-generated REST API over Postgres meant zero backend code for straightforward reads/writes, freeing the actual build time for the parts that mattered. The one deliberate exception — a serverless proxy for TMDB calls — existed only because the TMDB key couldn't ship to the browser, not because the app needed a "real" backend for its own logic.

**Sync: 4-second polling, not Realtime subscriptions.** Websocket connection lifecycle management (reconnects, stale connections, cleanup) is a real cost for a handful of friends checking a shared spin result. Polling gets the same user-facing effect — "everyone sees the same pick within a few seconds" — with none of that lifecycle to debug. This was a scope cut made consciously, not a missed opportunity: revisit only if the group ever gets large enough that polling load actually matters, which it hasn't.

**Duplicate movies: a client check plus a real DB constraint.** The client-side "already in the repo" check is a UX nicety — it makes the common case feel instant. But two people adding the same movie within the same second is a real race the client can't see coming, and a duplicate would silently double that title's odds on the spin wheel. `tmdb_id unique` at the database level is the actual backstop. This pattern — client check for feel, DB constraint for correctness — repeats through the whole project (duplicate names, duplicate emails, later on).

**Spin weighting: spotlight as a group gets 70%, not a per-movie multiplier.** A flat "spotlighted movies get 3x odds" multiplier means the *number* of spotlighted movies changes everyone else's odds unpredictably — spotlight one movie in a pool of 5 and it dominates; spotlight three and each gets diluted differently. Fixing spotlighted movies as a *group* at a flat 70% combined share, split evenly among however many are chosen, keeps the promise simple and constant: "spotlighted picks are heavily favored, together," regardless of how many you tap. (Full math tradeoff in `docs/adr/0001-spotlight-weighting.md`.)

**Synced reveal without Realtime: decide first, animate after.** The wheel's spinning client decides the winner *immediately* and writes it to `game_lobby` before its own animation even finishes. Every other client's next poll picks up the same pre-decided winner and plays the identical fixed-duration animation. Nobody needs a live connection to "watch the spin together" — they're all just independently animating toward an answer that was already fixed the instant the spin started. This is the single idea that makes polling-instead-of-websockets actually feel synchronous. (ADR 0002.)

**The wheel-landing gap, caught late.** The first shipped version of the wheel spun via a generic CSS loop and then *cut to a text reveal* — it never actually rotated to stop on the winning slice. Accepted at the time as a reasonable scope cut. A later design review correctly called this out as a real problem, not a cosmetic shortcut: it's the single most-demoed interaction in the app, and it was about to get a second reveal layer (curtains) built on top of a mechanic that didn't actually land anywhere. Fix: since the winner is already decided the instant the spin starts (see above), the exact landing angle is computable up front from the same slice-size math already driving the wheel's rendering — so the wheel now animates to a *specific known rotation*, not an open-ended spin. This is a good example of a decision that looked fine in isolation and only became visibly wrong once the next feature was about to stack on top of it.

**Arena's score cap: a soft backstop, not real anti-cheat.** `round_high_score`'s `CHECK` constraint blocks the obvious "type 999999 into dev tools" move, derived loosely from the game's real theoretical ceiling. It does not stop a scripted fast-tapper. That's a named, accepted gap — closing it for real would mean a server-authoritative game loop, which is disproportionate for a friend-group app where the actual threat model is "one person being silly," not "an adversary." Naming the limit of a defense explicitly, in the code and in the spec, is cheaper than pretending it's airtight.

## The cinematic redesign (Module 4) — three rejected wheel designs before the right one

This is worth telling in full because "iterate until it's right" is a process decision, not just a visual one.

**First attempt:** recolor the existing amber pie-slice wheel to the new charcoal/gold palette. Rejected on sight — low-alpha slices on near-black read as a blank disc, no per-slice content, "a black box."

**Second attempt:** add rotated, truncated movie-title text to each slice. Rejected immediately, and correctly — a wheel that size cannot hold a real title without mangling it into something unreadable.

**What actually solved it: research real wheels instead of inventing a third guess.** Looking at how actual roulette wheels and gamified reward-wheel apps (the Zepto/Blinkit style) solve this revealed the same lesson from two different domains: meaning never lives in on-wheel text. It lives in a small badge, and the wheel body itself stays plain. That reframing — "the poster carries recognition, nobody needs to read a label" — is what the current poster-badge wheel is built on. The lesson generalizes past this one component: when a design problem feels stuck after two honest attempts, the fix is often not a third guess but finding the existing convention that already solved this exact problem elsewhere.

## Splitting Module 5 after a structured critique

Module 5 was originally scoped as one unit (trailers + curtain reveal + stats page) with one shared checkpoint. Two dispatched review agents — one deliberately adversarial, one consolidating the findings into concrete fixes — caught that bundling three functionally unrelated features under one verification step broke the project's own established discipline ("one coherent thing per module"), and did so right when two schema changes made isolated verification *more* valuable, not less. Split into 5a/5b/5c with the same total scope. This is the clearest example in the project of using an adversarial review step deliberately, not just writing code and hoping it's fine — catching a process mistake before it compounded into three modules' worth of tangled testing.

## Module 7 — real accounts, the module with the most rejected first drafts

This module had more genuine back-and-forth than any other, because it touches identity, security, and every existing screen at once.

**The trigger wasn't "we should add login."** It was the user reviewing the live deploy and flagging three things simultaneously: nobody was ever required to prove who they were, a layout bug, and — the one that actually changed scope — anyone with the link could act as anyone. Investigating that third point surfaced a real, silent production bug underneath it: `movie_repo` had never had a `delete` policy at all, meaning Module 6's Remove feature had been broken in production since the day it shipped, with nobody noticing because nothing ever threw an error — RLS just silently denied it.

**Rejected: an open-signup model.** The first draft of this module let anyone request access with any email. That sounded reasonable until "a stranger who finds the link can get in, just by owning an email address" was said out loud instead of left implicit. Closed signup — the owner invites, nobody self-registers — was the one requirement that became non-negotiable the moment it was stated concretely instead of abstractly.

**Rejected: an app-level allowlist check.** The obvious naive design — a plain table of allowed emails the login screen checks before calling Supabase — was rejected specifically because Supabase Auth has its own address on the internet. A determined caller could hit Supabase directly and skip the app's UI-level check entirely. The fix moved enforcement *inside* Supabase's own machinery instead of bolting it on beside: inviting someone actually creates their Supabase account ahead of time, and the login call itself passes `shouldCreateUser: false`, so Supabase refuses to issue a link for anyone without a pre-existing account. This is the project's clearest instance of "where does real enforcement actually live" as a design question, not an implementation detail.

**Caught in self-review, not in implementation: the self-promotion hole.** The natural-looking RLS policy "a user can update their own profile row" only restricts *which row*, not *which columns*. Under that policy alone, any logged-in member could set `is_owner: true` on themselves via an ordinary update — RLS was never going to catch that, because RLS answers "is this your row," never "which columns are you touching." Closed with a `before update` trigger that rejects client-initiated changes to `is_owner`/`revoked` specifically, with a `service_role` exception so the app's own admin endpoints could still legitimately set them. The first draft of that trigger blocked those columns unconditionally — which would have also broken the app's own invite/revoke endpoints, since a Postgres trigger fires regardless of who's asking. Caught before it shipped, not after.

**The bootstrap chicken-and-egg.** Manage Invites only appears to a logged-in owner. Nothing can log anyone in until an account exists — including the owner's own. Solved with a one-off script run once, directly against the live project, outside the app entirely. Worth naming as its own decision: not every account-creation path needs to live inside the product. The very first account is legitimately a special case, and pretending otherwise would have meant building UI for something that only ever happens once.

**Revoke: paused, not deleted — decided at design time, revisited and confirmed months later under real pressure.** From the start, revoking someone was specified to block *future* access while leaving their past contributions untouched, the same way a removed friend's old messages don't vanish from a group chat. What wasn't decided at design time was what happens when a revoked person needs to come back. See the next section — that gap became a real, live incident.

## The invite-crash incident and what it actually taught

A raw Postgres error (`duplicate key value violates unique constraint`) reached a real user's screen in production. Root cause: the invite endpoint called Supabase's `inviteUserByEmail` — which silently *resends* a link for an already-invited, not-yet-confirmed person rather than erroring — and then unconditionally tried to insert a new database row, colliding with the row the first invite attempt had already created. First fix: detect that case and treat a second "Add" on a pending invite as a deliberate resend. That closed the crash, but not the deeper gap it exposed.

**The deeper gap, surfaced by a real support request:** a revoked friend couldn't be re-invited *at all* — re-adding them hit a hard, permanent block, with no path back except direct database editing. This is where "revoke is paused, not deleted" — decided in principle at design time — got its first real test. The actual decision, worked through in conversation rather than assumed: re-inviting a revoked person should **restore their exact same account and history**, not spin up a fresh one. A fresh account would leave their old contributions attributed to a display name that then blocks their own new signup, since names are unique — restoring in place has none of that self-inflicted mess. This is a good example of a principle that sounded settled on paper needing a second, harder pass once an actual person hit the edge of it.

**Alongside that fix, a structured audit, not just a patch.** Rather than fix the one reported symptom and move on, three research passes ran in parallel — invite/revoke state-machine edge cases, an error-handling audit across every auth-related file, and a concurrency/data-integrity pass — surfacing 21 distinct issues total. Compiled into a scored risk register (Likelihood × Impact) rather than an unordered bug list, specifically so severity and priority were visible at a glance instead of buried in prose. Of those 21: about ten were unambiguous bugs (raw errors leaking to users, an orphaned account left behind when an invite partially failed, no protection against revoking the last remaining owner) fixed directly without further discussion; a handful were genuine product decisions (the restore-vs-fresh-account question above, whether revoking a fellow owner needs an extra confirmation step) resolved through direct conversation; one or two were accepted, named limitations (a revoked email address later belonging to a genuinely different person is undetectable by the app — accepted rather than half-solved).

## The email-delivery saga — a good example of "the elegant fix" turning out to be the wrong question

Inviting a second real user hit Supabase's default email service's rate limit (2/hour) almost immediately — a limit explicitly documented as non-production-grade. The obvious-looking fix (personal Gmail SMTP via an app password) worked once, then silently stopped delivering with zero error and zero bounce. Chased that down through several real hypotheses before finding the actual cause:

- Considered and researched: Gmail's own anti-abuse systems discouraging personal accounts being used as an automated relay (a real, documented phenomenon).
- Considered and researched, in order: Mailgun's sandbox (rejected — capped at 5 explicitly-authorized recipients), Resend without a domain (rejected — can only send to yourself, not to anyone else, without owning a domain), SendGrid (rejected — its permanent free tier no longer exists as of 2025), Brevo (initially recommended, then *un*-recommended after actually checking: it mandatorily wraps every link through its own tracking domain, which is both a phishing-adjacent look for friends clicking an invite link and a real functional risk — automated link-safety scanners can "pre-click" a wrapped link and burn a one-time auth token before the real recipient ever sees it), and finally Amazon SES (the strongest technical answer — doesn't wrap links by default, genuinely free at this volume — but requires a credit card on file to create the account at all, a hard no given an explicit "not even if it's not charged" constraint).
- **The actual root cause, found only by checking the internal evidence instead of continuing to theorize externally:** the Gmail app password entered into Supabase's SMTP settings had spaces in it, copied verbatim from how Google displays it for readability. Removing the spaces fixed delivery immediately.

The real lesson here isn't "always check the simple thing first" as a platitude — it's that a plausible, well-researched external explanation (Gmail's abuse detection) can absorb a lot of genuine investigation time while the actual bug is sitting in a config field the whole time. The fix for that isn't giving up on research; it's treating internal, first-party evidence (Supabase's own auth logs, the actual SMTP config screen, the literal warning banner Supabase itself displayed) as higher-priority than external theorizing, and checking it earlier rather than later.

## The mobile layout — two bugs that looked like design flaws but were CSS defaults

**"Zoom is broken" turned out to be two separate, smaller things, not one big one.** The original complaint sounded like a single layout bug. Investigating it surfaced that it was actually two independent issues wearing the same symptom: a long vertical scroll on true mobile widths (a UX gap — the responsive grid itself was never broken), and a genuine CSS overflow bug inside each panel's own control row.

**The tab-switcher decision.** Once "mobile is roughly 99% of actual usage" was said explicitly rather than treated as a minor case, the fix stopped being "patch the zoom bug" and became "the primary usage pattern deserves a primary-usage-pattern layout." A lowfi wireframe comparison made the choice concrete before any code was written: keep the long scroll and just fix the narrow-viewport overflow, or replace the scroll with a Repo/Wheel/Arena tab switcher below the desktop breakpoint. The tab switcher won once the usage split was made explicit.

**A real regression, caught live, that's worth understanding precisely.** The first implementation of the tab switcher used the native HTML `hidden` attribute (driven by which tab was selected) plus a CSS class meant to unconditionally override it back to visible at desktop widths. The reasoning — that author-written CSS should beat the browser's own default styling for the `hidden` attribute — is correct in CSS theory. It did not hold up in the actual deployed build: at a wide desktop window, two of the three panels stayed invisible, leaving empty space where they should have been. Every test passed the whole time, for a specific, instructive reason: this project's test environment never loads real compiled CSS at all, so every test claiming to "confirm" the visual behavior was only ever checking the underlying DOM attribute directly, completely blind to whether any actual stylesheet override worked. The tests were internally consistent and green while testing the wrong thing. Fixed by using only Tailwind's own classes for both directions — the same single-origin mechanism already working correctly elsewhere in the same file — and this time verified by actually inspecting the compiled CSS output rather than reasoning about cascade order from first principles. **The general lesson, kept in the codebase as a comment so it doesn't get relearned the hard way:** a green test suite proves the logic is right; it does not prove the browser renders it the way you think, and those are different claims that need different verification.

**The genre-filter overflow, once actually diagnosed, was almost embarrassingly simple.** A `<select>` element has a browser-enforced minimum width, and flex items don't shrink below their own content size unless explicitly told to. A control row with no `flex-wrap` doesn't shrink or wrap when it doesn't fit — it just overflows its container, which is indistinguishable at a glance from "the layout is broken" but is actually one missing CSS property. Real production systems avoid this with two habits held as defaults, not afterthoughts: toolbar-style rows get `flex-wrap` by default, and layouts get resized/zoomed during development instead of eyeballed once at a single comfortable width. That second habit — not the first bug, and not the second one — is the actual root cause tying both mobile issues together: single-viewport testing during a fast, module-by-module build, never revisited with a dedicated cross-width pass until a real user hit the edges of it.

## Working practices used throughout, not just individual decisions

- **Spec-driven, not code-driven.** `spec.md` is the source of truth; code is generated from it, not the reverse. Every module's design gets written down, self-reviewed, and approved *before* implementation starts.
- **A living build tracker, not memory.** `TRACKER.md` is updated the same day something changes state, specifically so "what's actually done" never depends on anyone's recollection of a conversation.
- **Testing boundary, held consistently:** pure logic gets unit tests; user-visible behavior gets component tests queried by role/text, never by class name or color; thin network wrappers around Supabase calls are deliberately *not* unit tested — only the judgment-call logic extracted out of them is. This boundary is what made "the tests were green but the browser was wrong" (above) legible as a specific, nameable gap instead of a vague trust problem.
- **Structured, adversarial review as a real step, not a formality.** Twice in this project (the Module 4/5 design review, and tonight's three-pronged data-integrity audit), dispatching multiple independent review passes surfaced real problems a single continuous train of thought had missed — not because anyone was careless, but because a single thread of reasoning has blind spots a second, differently-angled one doesn't.
- **Obvious bugs get fixed directly; product decisions go through a real conversation.** This distinction — established explicitly after an early misstep where two UX fixes were shipped without alignment — is the operating rule for everything after Module 7: a raw error leaking to a user is a bug, fixed on sight; whether a revoked person's old account should be restored or replaced is a product decision, worked through in the open before a line of code changes.
